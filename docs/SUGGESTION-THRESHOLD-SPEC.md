# ImageAll 建议阈值（用户可调）规格

> 状态：Approved（ST-P1–P8 由所有者全部按建议批准，2026-07-22）  
> 日期：2026-07-22  
> 基线：`main@81a3201c601c662d067ab73e70eec1ef83bb2548`（撰写时；实施开工前以文档提交后的精确 HEAD 为准）  
> 相关决策：ADR-040（本能力）；ADR-039（三轨并行）；`TRAINING-WORKSPACE-SPEC.md`  
> 角色边界：本文定义产品语义、契约、验收门与切片；可执行实现另开切片，不得与无关脏改动混提

## 1. 目标

把三条个人建议路径上的「进队门槛」升级为一等**用户可感知、可调节**的产品设置：

1. **阈值属于用户**：算法不得在后台静默改写生效阈值；
2. **三轨分立**：`featureKnn` / `personalCentroid` / `personalAdamW` 分数语义不同，阈值互不可比、互不共用；
3. **可感知**：用户能在设置与审核场景中看到「当前门槛是什么、分数相对门槛如何」；
4. **可调节**：用户可在设置中改默认值，也可按标签覆盖；改完后下一次 Top N 生成立刻遵守。

本规格不改变：原图只读、人工事实优先、真实测试数据保护、默认不 push。  
本规格不把阈值校准伪装成「模型自动调参」。

|---|---|---|---|

| **ST-P4** | Review 队列行**展示分数**（与 origin 徽章并列），便于用户对照门槛；废止训练工程规格 §5.3「首版不展示 score」对本能力的约束 | 采纳 | **已批准**（2026-07-22） |

| **ST-P7** | 分数展示用**原始 score + 方法说明**，不做伪百分比；帮助文案写明「三轨分数不可横向比较」 | 采纳 | **已批准**（2026-07-22） |
| **ST-P8** | 重建/训练模型**不自动改阈值**；用户可手动重调或点「参考建议」后再确认 | 采纳 | **已批准**（2026-07-22） |

未纳入本轮：自动按接受/拒绝分布静默改门槛、跨设备同步阈值、标准公共模型 `autoAssigned` 门槛、准确率仪表盘。

## 3. 用户心智模型

```text
某次「更新 XXX Top 100」
  → 只扫描「该标签尚无人工决定」的候选
  → 打分
  → 保留 score > 生效阈值 的候选
  → 再按分数取 Top 100（不够 100 不强行凑）
  → 写入该轨待审队列
```

用户调节的是「**多高分才值得我审核**」，不是「模型内部 loss」。

### 3.1 方法产品名与 method 键

| 产品名 | method 键 | 当前生成入口 |
|---|---|---|
| 特征向量 | `featureKnn` | 「更新特征向量 Top 100」 |
| 个人模型 | `personalCentroid` | 「更新个人模型 Top 100」 |
| 超级个人模型 | `personalAdamW` | 「更新超级个人模型 Top 100」 |

## 4. 信息架构与 UI

### 4.1 App Settings（主入口）

在现有 `Settings`（`AppModelSettingsView`）增加独立分组，不塞进「本地模型」开关里：

```text
本地模型
  启用 DINOv2 Small …
建议阈值                          ← 新 Section
  说明：三轨分数含义不同，请分别调节；默认 0 表示只要正分就可进队。
  特征向量默认门槛     [滑块/步进]  当前值：0.00
  个人模型默认门槛     [滑块/步进]
  超级个人模型默认门槛 [滑块/步进]
  按标签覆盖…          → 打开标签覆盖列表（可搜索）
```

标签覆盖列表：每行一个标签名 + 三列可编辑门槛（空 = 继承默认）。支持「清除覆盖」。

### 4.2 标签待审核面板（副入口）

在「生成/更新 Top 100」确认区或邻近控件展示本标签三轨生效阈值（已解析覆盖），可就地编辑；写入规则与 Settings 相同。

生成确认文案必须带上生效阈值，例如：

> 将用超级个人模型扫描…，只保留分数 **高于 0.12** 且最高的 100 条。

### 4.3 Review 队列行（ST-P4）

每行展示：

- origin 徽章；
- **分数**（保留合适小数位，有限值）；
- 可选：相对门槛的弱提示（例如低于当前门槛的行不应再出现；刷新动作后保证）。

排序仍：先 origin 优先级（与 ADR-039 / 训练工程规格一致），再 `score DESC`。

### 4.4 「参考建议值」（可选辅助，非自动生效）

允许在某标签、某方法上提供只读参考值（例如基于最近一次审核中「拒绝」分数的高分位），UI 文案必须是：

> 参考建议：0.18（根据近期拒绝样本估算）。是否采用？

用户点「采用」才写入覆盖；点「忽略」不改库。首版可延后到 ST3，但契约预留，禁止实现成静默写入。

## 5. 数据契约

建议独立 migration（**v015**，不绑死未完成的 v014 训练工程切片；若 v014 尚未合入生产路径，实施时以当时 `CatalogMigrationID.knownOrdered` 末项 +1 为准）。

### 5.1 `suggestion_score_threshold_default`

| 列 | 约束 |
|---|---|
| `method` | PK；`featureKnn` \| `personalCentroid` \| `personalAdamW` |
| `min_score` | REAL NOT NULL；有限值 |
| `updated_at_ms` | 非负整数 |

迁移时插入三行，`min_score = 0`。

### 5.2 `suggestion_score_threshold_override`

| 列 | 约束 |
|---|---|
| `tag_id` | TEXT UUID，FK → `tag(id)` ON DELETE CASCADE |
| `method` | 同上三枚举 |
| `min_score` | REAL NOT NULL；有限值 |
| `updated_at_ms` | 非负整数 |
| PK | `(tag_id, method)` |

### 5.3 解析 API（Application 端口）

```text
effectiveMinScore(tagID, method) -> Double
  = override(tagID, method) ?? default(method) ?? 0

setDefault(method, minScore)
setOverride(tagID, method, minScore)
clearOverride(tagID, method)
```

写入校验：`minScore.isFinite`；不强制区间（各方法分数分布不同），UI 可用宽松滑块范围 + 数字输入。

### 5.4 与既有 `tag_model_revision.threshold` 的关系

Feature Print 历史表已有 `threshold` 列，当前实现写入多为 `0`。  
**本规格权威阈值表是 5.1/5.2**；生成 Feature Print Top N 时读 effectiveMinScore，**不再**把用户设置写死进不可变 revision 当作唯一真相。  
可选：新生成的 revision 快照里把当时 effective 值抄进 `threshold` 仅作审计，不以它覆盖用户设置。

### 5.5 生成与刷新语义

**生成 Top N（三条路径共用）：**

1. 候选：无该标签人工决定的资产（已有契约）；
2. 打分；
3. 过滤：`score > effectiveMinScore(tag, method)` 且有限；
4. 排序取 Top `maxPendingSuggestionsPerTag`（100）；
5. 写入对应 prediction 表 / 槽。

**显式「按当前阈值刷新本标签待审队列」（ST-P5）：**

- 仅删除该标签、该 method 下 `score <= effectiveMinScore` 的 pending；
- 不重新扫描图库、不重算 embedding；
- 不触碰人工决定与其它 method 的 pending。

## 6. 与训练工程规格的交叉修订

`TRAINING-WORKSPACE-SPEC.md` §5.3「首版仍不展示原始 score」在 **ST-P4 批准后**改为：

- 训练工程 T3 切片：队列行**必须**展示 score（服务阈值校准）；
- 对比仍可靠徽章；分数用于「是否高于我的门槛」，不作跨轨概率比较（ST-P7）。

ADR-039 全并行身份不变。

## 8. 测试矩阵

1. 默认 0：行为与当前「正分进队」一致（回归）；
2. 提高某方法默认后，该轨新生成不再写入 `score <= 新默认` 的行；其它方法不受影响；
3. 标签覆盖优先于默认；清除覆盖后恢复默认；
4. 删除标签级联删除覆盖行；
5. 「刷新待审」只删低于门槛的 pending，人工决定与其它轨保留；
6. Settings 改值进程重启后仍在；
7. UI：确认对话框展示生效阈值；队列行可见分数；
8. 反例：算法路径不得在无用户确认时 `UPDATE` 默认/覆盖表。

## 10. 风险

| 风险 | 缓解 |
|---|---|
| 用户把三轨分数当同一尺度比较 | ST-P7 文案 + Settings 说明 |
| 提高门槛后队列变空，误以为 bug | 生成完成 notice 写明「高于阈值 N 条 / 候选 M」 |
| 与 v014 训练工程 migration 顺序冲突 | 阈值用独立下一 migration；不改写已批准 migration |
| Feature Print 旧 `threshold` 列语义混淆 | §5.4：用户设置表为权威 |
