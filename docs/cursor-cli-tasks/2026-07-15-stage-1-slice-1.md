### 5.2 阻塞项

1. **原子性证据没有走生产实现。**
   `testCreateAndApplyRollsBackWhenDecisionPhaseFails` 调用测试文件内的
   `FailingAfterTagInsertRepository`，该类型复制一套 INSERT/transaction 逻辑后主动抛错，
   没有调用生产 `GRDBTagCatalogRepository.createTagAndApply`，所以只能证明测试假实现会回滚。
   `testChunkedBatchRemainsSingleTransaction` 也只证明 1,200 条成功写入，没有证明后段 SQL
   失败时早期块不会残留。
2. **强制测试矩阵缺口。** 缺少 `file_name` NUL 反例；四种 Source state 中只覆盖
   disabled/unavailable；source OR 测试选择了 fixture 的全部来源，不能证明未选择来源被排除；
   search 的 `%`/`_`/escape 只查询一个不存在字符串，不能证明字面量命中且不能证明注入无效；
   排序没有 duplicate time、NOCASE collision、null file name 的明确期望；分页只遍历 newest，
   未遍历 oldest/fileNameAscending。
3. **create-and-apply 的返回契约不足。** 输入没有 Tag ID，当前返回值也不含新建 `Tag`
   或其 ID；上层不能在不重查、不猜测的情况下刷新新 Tag 并构造
   `TagMutationPriorStateSnapshot`，与“从成功命令返回前态并可恢复”的契约不闭合。
4. **持久化错误可越过 Application 端口。** query/tag repository 多个公共方法会直接抛出
   GRDB `DatabaseError`；该错误的描述可能含 SQL，既泄漏 Infrastructure 类型，也违反
   `persistenceFailure` 和无 SQL/用户数据错误语义。所有公共端口必须保留已定义的
   `CatalogQueryError`，把其他底层错误收敛为 `.persistenceFailure`。
5. **schema introspection 尚未锁死表达式顺序。** 当前测试只统计 unnamed expression 数量并
   搜索若干 DDL 片段；调换两个 expression、改变它们和 named key 的相对顺序，测试仍可能通过。
   必须从真实 `sqlite_schema`/`index_xinfo` 校验最终 key 顺序、collation 与完整 partial predicate，
   不能读取生产 Swift 常量作为 expected。
6. **生产代码含仅供测试的 v001 migrator 工厂。** `CatalogDatabase.makeV001OnlyMigrator()`
   只被测试调用，应移到 `ImageAllTests`，生产 `CatalogDatabase` 只保留正式 migration 链。
