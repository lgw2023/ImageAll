## 6. Codex 初审结论与返修要求

### 6.1 结论

`02f0003` 保持了 v001/v002、App/SwiftUI 与切片停止边界，v003 DDL 的主体形状也符合交接单；但实现与测试没有达到第 5～12 节的安全及行为门。529 项绿灯中，新增 30 项远不足以证明交接单明确要求的反例、fault matrix、路径对抗和像素行为，并出现了“测试名声称 LRU、实际不触发淘汰”的弱化断言。切片 4 不通过。

### 6.2 阻塞项

1. **有效 cache hit 被生成资格错误阻断。** `loadOrGenerate` 在查 exact-key entry 前先要求完整 generation context 且 `source.state = active`、Asset `available`；因此 Source 离线、disabled 或 authorizationRequired 时，即使同 revision 已有完整有效对象也无法返回。应先取得 Asset 当前 revision 并验证 exact hit；只有 miss/损坏后才进入新生成资格与授权流程。增加“生成后 Source 切换四种非 active 状态仍命中且 scope 不启动；同状态 miss 不生成”的运行测试。
2. **缓存文件操作没有满足目录句柄锚定/no-follow 合同。** `Data(contentsOf:)`、绝对路径 `rename`/`unlink`、`FileManager.enumerator` 和只检查 `versionRoot`/父目录会留下 staging、objects、shard 祖先被 symlink/rename 置换的 TOCTOU 窗口；尤其 `unlink` 会跟随被替换的父目录而可能删除派生根外同名文件。所有读、排他创建、sync、rename、删除和维护遍历必须锚定已打开且验证过的自有目录 FD，以相对组件和 no-follow 等价语义执行；逐层拒绝 root/staging/objects/shard/object symlink、未知层级及外部目标。测试必须放置外部哨兵并证明零触碰，不能只测 versionRoot symlink。
3. **source fingerprint 不是从已打开对象取得。** source bytes 从安全 FD 读取，但 size/mtime/resource ID 随后用拼接 URL 查询；post check 也重新按路径查询，不能证明“同一打开对象”稳定，存在 locator 替换窗口。改为对打开 FD `fstat`/等价句柄事实做 pre/post，再独立 no-follow 重开 current locator 比较 resource identity；最终 DB 事务继续重验证。用可控 seam 覆盖解码期间 size、mtime、resource ID、locator、revision、availability、Source state 和 fingerprint 分别变化，断言零 entry/零本次对象且事实不变。
4. **发布顺序与 fault coverage 不完整。** staging 写完后当前代码只重新校验内存 artifact，没有从 staging 文件复读 size/hash/解码；普通 `rename` 允许覆盖已存在目标。必须从 staging FD 复读验证，并采用“目标存在则失败”的同卷原子发布语义。逐一实际触发 staging create/write/sync/validate、rename、rename 后 DB 前、DB insert/update、final revalidation、LRU touch、替换旧 key 事务和旧对象删除 fault；比较稳定排序的完整 Asset/fingerprint/cache-entry 事实与缓存树。所有公开端口只泄出封闭 `DerivedImageError`，不得透传 `DerivedImageSecureIOError`、GRDB error 或带路径的底层 error。
