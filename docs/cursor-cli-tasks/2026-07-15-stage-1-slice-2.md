### 4.5 Codex 二审结论：未通过

`3859998` 已保持切片 2、UI 零接线和真实数据零访问，首次十项阻塞中的多数行为也已收敛；但独立源码与测试矩阵复审仍发现以下同一交接单阻塞：

1. **未取得 scope 后继续探测 URL**：普通访问的 `startAccessing` 返回 false 后，`classifyScopeStartFailure` 调用 `checkResourceIsReachable()`。这违反交接单 §7.2 的“start false 不执行受保护操作”。该路径没有已取得的明确离线证据，必须保守写 `authorizationRequired`，且用 closure 未执行、start 1 次尝试/stop 0 次、事实保留测试证明。
2. **普通访问状态矩阵仍不完整**：补充确定性的 `NSFileNoSuchFileError`/`ENOENT` bookmark resolve 反例，证明明确离线写 `unavailable`；补充 scope 拒绝和已取得 scope 后根失效/不可读路径，证明写 `authorizationRequired`，closure 不执行，根失效路径 start/stop 各一次。状态写失败继续为 `persistenceFailure`，全部事实保留。
3. **停用强制矩阵未真实覆盖**：当前测试只构造 pending 与两种 running；必须实际构造并逐字段断言 pending、paused、retryableFailed、running(none/pause)、terminal 和其他 kind。前三类 cancelled 并清 control/lease/error；running 仅提升 cancel；terminal/其他 kind 不变。
4. **停用回滚故障点过早**：`installDisableJobConvergenceAbortTrigger` 在 Source state 更新时立即失败，没有证明“先改 Source、再取消一批 Job、后段 running Job 更新失败”能整体回滚。故障必须设在后段真实 Job SQL，并断言 Source 及此前/当前全部 Job 字段恢复原值。
5. **重新授权回滚故障点过早**：当前 trigger 同样在 Source 更新时失败。无活跃 Job 场景应在后续 Job INSERT 上制造真实 SQL failure，证明已执行的 bookmark/display name/state 更新与 Job 集合全部回滚。
6. **重新授权事务缺少 state 再验证**：coordinator 在显示选择器前检查 state，但 repository 事务只检查 kind；选择器等待期间若 Source 被停用，事务仍会重新激活它。repository 必须在同一事务内再次要求 state 为 `unavailable` 或 `authorizationRequired`，否则结构化拒绝且字段/Job 不变；增加能证明该竞态边界的测试。
7. **异常行被静默跳过**：`fetchAllFolderSources` 使用 `compactMap`，异常 folder row 会从重叠审计中消失。改为任一无法解码的 folder row 都收敛为 `persistenceFailure`，不得当作不存在。

### 4.2 Codex 初审结论：未通过
