# ImageAll 本机真实测试数据安全策略

> 状态：Active<br>
> 生效日期：2026-07-14<br>
> 适用对象：Codex、Cursor、人工开发者以及任何自动化工具<br>
> 目的：允许未来用真实照片做只读验证，同时防止开发、测试和清理操作损坏用户资产

## 1. 受保护数据范围

项目所有者已指定以下真实用户数据可以在未来适当阶段作为只读验证来源：

1. Apple Photos Library：`/Volumes/HDD2/Photos Library.photoslibrary`；
2. 普通文件夹：`/Volumes/HDD2` 顶层所有名称以四位 ASCII 数字年份开头的真实目录。

第二项是匹配规则，不把当前个人目录名称复制进仓库。新增的同类年份目录自动受到同等保护；符号链接、Finder alias 及其目标不会仅因链接名称匹配而自动取得访问授权。`/Volumes/HDD2` 上不匹配该规则的其他内容仍未获项目访问授权。

这些路径包含用户真实照片，不是可丢弃 fixture、临时目录、备份工作区或故障注入对象。“可作为测试数据”不包含删除、修改、迁移、修复或重新整理这些资产的许可。

## 2. 按阶段划分的使用边界

### 阶段 0

- 不扫描、不解码、不索引上述真实照片；
- 实现者、自动化测试和阶段 0 验收命令都不得访问上述路径；如以后确需核对挂载或路径存在性，必须作为单独的只读环境审计取得项目所有者授权，不能夹带在切片实施中；
- 数据库、migration、Repository、Job、快照和启动测试只使用各测试创建的独立临时目录；
- 不在生产代码、测试代码或 Xcode 配置中硬编码本机绝对测试数据路径；

### 阶段 1：文件夹来源

年份文件夹只能作为额外的人工只读 smoke 数据源，并且必须同时满足：

1. 对应切片已明确授权真实文件夹接入；
2. 用户通过系统目录选择器主动选择来源；
3. 应用使用只读 security-scoped bookmark；
4. 自动化测试继续使用合成 fixture，不依赖 `/Volumes/HDD2`；
5. 验收证明来源内没有新增、修改、移动、重命名或删除任何文件。

### 阶段 2：Apple Photos 来源

`Photos Library.photoslibrary` 的绝对路径只用于标识受保护资产，不是应用的文件系统输入。必须遵守：

- 只能通过 PhotoKit 和用户授权读取 Photos 资产；
- 不得遍历、解析或依赖 `.photoslibrary` 包内部结构；
- 不得由工具自动打开、切换、修复或设为 System Photo Library；
- 是否切换或打开该图库由用户在 Photos 中显式操作；
- 不调用删除、编辑、导入、关键词写入或其他 Photos 写入 API；
- 对受保护真实图库的 smoke 默认禁止触发 iCloud 网络下载；如某项验收必须下载 iCloud-only 资源，须取得单独的 `CloudDownloadGrant` 和用户确认，并在开始前说明 Photos 可能产生系统管理的本地缓存与磁盘占用；
- PhotoKit 没有供本产品使用的独立只读 access level，因此 `.readWrite` 授权不等于写入许可；评审必须证明生产 target 没有 Photos mutation API 调用路径；
- 专门的可丢弃测试 Photos Library 仍用于故障和边界测试；此真实图库只作为额外的只读人工 smoke 来源。

## 3. 禁止操作

在没有项目所有者针对具体操作给出新的书面授权前，禁止对受保护数据执行或让应用执行：

- 删除、移动、重命名、覆盖、替换、去重或原地转换；
- 创建 sidecar、隐藏文件、缩略图、索引、数据库、缓存或临时文件；
- 修改权限、所有者、扩展属性、时间戳或其他文件元数据；
- 使用带删除语义的同步、清理或镜像操作；
- Photos 中的删除、编辑、导入、整理、修复或关键词写入；
- 把受保护路径或其父目录作为测试 teardown、缓存淘汰或临时目录清理目标；
- 为方便测试而批量复制几个 TB 的原始档案。

任何需要验证删除、移动、权限错误、文件损坏或图库修复的测试，都必须使用独立创建、明确可丢弃的小型 fixture 或专用测试图库。

## 4. 允许的数据流

```text
受保护真实照片
        |
        | 只读、用户授权、对应阶段人工验证
        v
ImageAll 解码/索引流程
        |
        +--> App 容器内事实库
        +--> App Caches 内可重建派生数据
        +--> 聚合测试证据

禁止任何反向写入受保护来源
```

运行日志和评审证据不得记录图片内容、bookmark 数据、Photos local identifier 或逐项完整路径。可以记录聚合数量、耗时、错误码、来源类型以及是否通过只读安全检查。

## 5. 清理与故障测试安全门

实现者必须把“应用拥有的测试产物”和“用户来源”作为不同信任域：

- teardown 只清理当前测试创建并登记的唯一临时目录；
- 缓存清理只作用于 App Caches 中由 ImageAll 创建的内容；
- 数据库恢复只作用于 App 容器内目录库及其快照；
- 清理前必须解析并验证规范路径、符号链接和实际允许根目录归属；不能仅凭名称、后缀或调用方传入路径判断，并且验证与操作之间必须避免可被路径替换绕过的窗口；
- 目标等于受保护路径、位于其内部、是其祖先目录，或安全归属无法证明时，操作必须拒绝并返回结构化错误。

## 6. 授权规则与交接检查

以下措辞不构成对真实照片的修改授权：

- “继续下一切片”；
- “运行完整测试”；
- “清理测试产物”；
- “使用这些照片作为测试数据”；
- 对某个目录授予读取权限。

改变只读策略需要项目所有者明确指出具体路径、具体写操作、目的、回滚与备份条件，并由 Codex 先更新相应架构/交接规格。授权必须记录在版本化的阶段交接单或 ADR 中，默认只对所列操作和该次任务有效，任务结束即失效；不能从旧授权推导新操作。当前不存在任何受保护真实照片写入授权。

`/Users/liguowei/Downloads` 是项目所有者于 2026-07-15 单独批准的低风险本机测试来源，不属于第 1 节的 `/Volumes/HDD2` 受保护档案。本阶段可由用户通过系统选择器连接并用于只读运行 smoke；当前产品仍是只读来源模型，因此这项批准不改变生产代码的源端零写入约束，也不需要为验证而主动删除或损坏 Downloads 中的文件。若后续确需验证删除、覆盖或移动能力，应先限定具体样本和动作，避免把整个 Downloads 作为清理根目录。

Cursor 在涉及文件夹或 Photos 的切片开工和交回时，必须分别声明：

1. 是否访问了受保护数据；
2. 使用的是自动化 fixture 还是人工真实数据 smoke；
3. 是否发生任何源端写入；预期答案必须为“否”；
4. 测试产物实际写入的应用拥有目录；
5. 使用何种证据证明照片未被删除、移动、重命名或覆盖。

每个涉及真实数据的阶段交接单还必须定义该阶段可负担、保护隐私的证明方法。至少包括生产代码中没有源端写 API 的静态检查、运行期间写操作计数为零，以及只报告聚合数量和错误码的人工 smoke 记录；不能仅以“测试没有报错”作为未写入证明，也不能为了取证对几个 TB 的来源执行全量 hash。
